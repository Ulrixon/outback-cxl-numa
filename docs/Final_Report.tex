\section{Introduction}
% ══════════════════════════════════════════════════════════════════════════════

\subsection{Background}

Outback~\cite{outback} is an RDMA-based Key-Value Store designed for disaggregated memory architectures. It employs a disaggregated memory model where Memory Nodes (MN) store KV data and Compute Nodes (CN) maintain a lightweight Dynamic Minimal Perfect Hashing (DMPH) index locally. The DMPH index (composed of Ludo~\cite{ludo} seed arrays and an Othello bucket locator) enables clients to resolve key locations with a single one-sided RDMA read, avoiding the CPU overhead of two-sided RPC on the memory node. The original paper claims up to a 5.03$\times$ throughput gain over baselines (RACE~\cite{race}, RPC-MICA~\cite{mica}) and a minimal memory footprint (12.5\,MB to index 20M KV pairs). When the hash table exceeds its load factor threshold, a resize operation rebuilds the DMPH index and redistributes data.

Compute Express Link (CXL)~\cite{cxl} is an open interconnect standard built on PCIe that enables cache-coherent shared memory between host CPUs and attached devices. CXL.mem provides load/store semantics to remote memory at sub-microsecond latency ($\sim$150--300\,ns), significantly faster than RDMA ($\sim$2--5\,$\mu$s) but slower than local DRAM ($\sim$100\,ns).

\subsection{Project Goal}

This project has two objectives:
\begin{enumerate}[nosep]
  \item \textbf{Reproduction:} Reproduce the original Outback RDMA experiments on CloudLab Apt \texttt{r320} nodes using the same Mellanox MX354A ConnectX-3 InfiniBand FDR (56\,Gbps) hardware as the paper's Figure~10 cluster, and analyze discrepancies against the paper's published results (Figures~9--17).
  \item \textbf{Extension:} Port Outback to a CXL-based platform by emulating CXL memory disaggregation through NUMA inter-socket shared memory on a CloudLab c6420 node, and compare the results against both the paper and the RDMA reproduction.
\end{enumerate}

Since dedicated CXL hardware was unavailable, we emulated CXL memory disaggregation using \textbf{NUMA inter-socket memory}. On a dual-socket machine, one NUMA node's memory serves as the ``remote'' Memory Node, while CPUs on the other socket act as Compute Nodes accessing that memory across the inter-socket UPI interconnect. This provides load/store semantics with $\sim$100--200\,ns cross-socket latency --- a reasonable proxy for CXL.mem behavior.

{\scriptsize $^a$In CXL/NUMA, the ``memory node'' is NUMA node~1's DRAM only~--- no MN CPU runs. The \texttt{mem\_threads} parameter is repurposed as \textbf{insert-lane sharding}: it partitions the \texttt{lane\_next\_free\_index} atomic allocation counter into $N$ independent lanes so that concurrent insert threads contend on different cache lines. It has zero effect on the read path (\texttt{ludo\_lookup} $\to$ \texttt{packed\_data} dereference never touches the counter) and negligible effect on writes because the bottleneck is UPI memory bus bandwidth for the actual data write, not the counter CAS. We expect our RDMA reproduction to show \textbf{lower aggregate throughput} than the paper's CX-6 results because of both the $\sim$2$\times$ slower NIC link (56\,Gbps FDR vs.\ 100\,Gbps CX-6) and the smaller CN count, but \textbf{comparable per-MN-thread throughput} once the MN CPU becomes the bottleneck (since serialization of RPC dispatch on the MN dominates over network bandwidth at small thread counts). Because we use the \emph{same} \texttt{r320} hardware as the paper's Figure~10 cluster, our per-MN-thread comparison against Fig.~10 is a like-for-like measurement. The CXL/NUMA extension should show \textbf{dramatically higher throughput and lower latency} by eliminating the network stack entirely.

\section{Findings: Build Effort and Issues}
% ══════════════════════════════════════════════════════════════════════════════

This section documents the practical challenges encountered in setting up and running the experiments. These issues are not discussed in the original paper but significantly affect reproducibility.

\subsection{RDMA Setup Challenges}

Getting the original Outback RDMA code to compile and run required resolving several hardware and software compatibility issues:

\textbf{1.\ Mellanox OFED Driver Mismatch.}
The Outback codebase depends on Mellanox's \texttt{libibverbs} and \texttt{libmlx4} libraries for ConnectX-3 RDMA operations. The default OFED driver version installed on CloudLab's Ubuntu 20.04 images was incompatible with the CX-3 firmware. We had to downgrade to MLNX\_OFED 4.9-x (the last release supporting CX-3 on kernel 5.4), as newer OFED versions dropped ConnectX-3 support entirely. This required manually removing the pre-installed driver stack and installing the legacy version:
\begin{lstlisting}[language=bash]
sudo apt-get remove -y ibverbs-providers libibverbs-dev
wget https://content.mellanox.com/ofed/MLNX_OFED-4.9-x.x.x/...
sudo ./mlnxofedinstall --force
sudo /etc/init.d/openibd restart
\end{lstlisting}

\textbf{2.\ Linux Kernel Version Incompatibility.}
The Outback build system uses R2 (a custom RDMA communication library) which relies on kernel-specific RDMA interfaces. Initial attempts on Ubuntu 22.04 (kernel 5.15) failed with compilation errors in the \texttt{xcomm} transport layer due to deprecated \texttt{ib\_*} API changes. We ultimately settled on Ubuntu 20.04 with kernel 5.4, which matches the kernel-era assumptions in the R2 library.

\textbf{3.\ NIC Port Index Hardcoding.}
The source code hardcodes the NIC port index used for RDMA operations. On our CloudLab CX-3 nodes, the active port varies depending on the physical NIC wiring. We found that the default \texttt{nic\_idx=0} parameter did not always correspond to the correct port. Discovering the correct port required running \texttt{ibv\_devinfo} on each node and then passing the correct \texttt{--nic\_idx} flag. On some nodes, only port 1 (not port 0) was connected, requiring a code-level change in the transport initialization.

\textbf{4.\ Finding Compatible CloudLab Hardware.}
Not all CloudLab node types support RDMA. We initially attempted to use c6420 nodes (which lack RDMA NICs) and d6515 nodes (which have ConnectX-5/6 but required different OFED versions). Only nodes carrying the legacy ConnectX-3 \texttt{mlx4} hardware worked with the OFED driver and code shipped by the artifact. Securing 3 simultaneous CX-3 nodes on CloudLab required scheduling around availability windows.

\textbf{5.\ Compilation and Dependency Issues.}
The build system expects specific versions of \texttt{cmake} ($\geq$3.16), \texttt{boost} (1.65+), and \texttt{gflags}. Several header files contained implicit dependencies on internal Mellanox headers (e.g., \texttt{hrd.h} from the MICA subsystem) that were not documented. The \texttt{setup-env.sh} script partially automates dependency installation but does not handle the OFED driver.

\subsubsection{Transport Layer: RDMA $\to$ POSIX Shared Memory}
The RDMA server registers memory regions via Mellanox NIC and serves them through Queue Pairs using \texttt{UDTransport}. In the NUMA port, \texttt{shm\_open()} creates files in \texttt{/dev/shm/}, \texttt{ftruncate()} sizes them, and \texttt{numa\_alloc\_onnode()} binds pages to NUMA node~1. A \texttt{SharedNumaRegistry} metadata structure stores server state, resize epochs, client liveness, and allocation lanes. Clients call \texttt{mmap()} to map the server's memory directly --- emulating CXL.mem load/store semantics.

\subsubsection{Execution Model: Coroutines $\to$ Synchronous Batched Loops}
RDMA's $\sim$2--5\,$\mu$s round-trip justified the R2 coroutine scheduler. With NUMA cross-socket latency at $\sim$100--200\,ns, coroutine context-switching ($\sim$50--100\,ns) would exceed the access time. All coroutine logic was removed; \texttt{numa\_search()} executes as a direct pointer dereference.

\subsubsection{Concurrency: Queue Pairs $\to$ Software Lanes}
Without NIC hardware, concurrent \texttt{\_\_sync\_fetch\_and\_add} operations cause cache-line bouncing. Setting \texttt{-{}-mem\_threads=N} creates $N$ independent atomic counters; each thread selects its lane via \texttt{thread\_id \% N}.

\subsubsection{Resize State Machine}\label{sec:resize-sm}
A six-state orchestrator was implemented:
\begin{center}
\small
$\text{NORMAL} \to \text{PRE\_RESIZE} \to \text{COPYING}$ \\
$\to \text{READY\_TO\_SWITCH} \to \text{DRAIN\_OLD} \to \text{GC\_PENDING} \to \text{NORMAL}$
\end{center}
Each epoch creates a new versioned shared-memory region. Clients detect changes through a generation counter and transparently remap. Structural mutations are deferred during COPYING/DRAIN and replayed after the switch.

\subsection{Codebase Issues Discovered}\label{sec:codebase-issues}

During reproduction, we discovered several undisclosed codebase limitations:

\begin{itemize}[nosep]
  \item \textbf{Resize on separate branch:} The resizing code was found on a separate Git branch with hard-coded 30-second benchmark limits. The standard main-branch tests run \textbf{without resizing enabled}, drastically altering the performance profile compared to the paper's Figure~17.
  \item \textbf{Broken zipfian for datasets:} Zipfian distribution for FB/OSM datasets was non-functional in the source code, despite the paper showing zipfian results (Figure~11c,d). The code accepts zipfian parameters but only applies them to YCSB workloads.
  \item \textbf{Memory requirements undisclosed:} 80M KV pairs causes an OOM on the r320's 16\,GB DRAM (the raw KV array plus the DMPH index plus the pre-registered RDMA huge-page MR exceed available memory), while the paper tests 80M on r650 (256\,GB) without disclosing this footprint.
  \item \textbf{No DMPH memory monitor:} The source lacked any way to measure the DMPH index size that the paper prominently reports in Figure~16. We developed a custom monitor from scratch.
  \item \textbf{Narrow hardware compatibility:} The paper does not disclose the strict driver/kernel/NIC requirements described in \S3.1.
  \item \textbf{YCSB-E (range scan) unimplemented:} We attempted to evaluate YCSB-E (95\% Scan, 5\% Insert), which tests range query performance --- a critical workload for real applications. The original source code does not implement range scan operations at all, and the paper does not report YCSB-E results. This is a significant omission: DMPH-based indexing provides $O(1)$ point lookups but has no inherent support for ordered iteration, making range scans fundamentally difficult for Outback's architecture.
\end{itemize}

\subsubsection{Comparison to Paper Figure~10 (CX-3 cluster)}

\section{Stress Tests and Breaking Points}
% ══════════════════════════════════════════════════════════════════════════════

Beyond reproducing the paper's claimed results, we deliberately tested the system under conditions that push it past its designed operating envelope. These experiments reveal fragility not captured by the paper:

\begin{enumerate}[nosep]
  \item \textbf{OOM at 80M keys (RDMA):} The r320's 16\,GB DRAM per node is insufficient for 80M KV pairs once the raw KV array, the DMPH index, and the pre-registered RDMA huge-page MR are all in memory simultaneously. The server process crashes with an out-of-memory error during bring-up. The paper tests 80M on r650 (256\,GB) without disclosing this constraint.
  \item \textbf{Memory swapping at 60M keys:} At 60M keys with LF=0.80, throughput collapsed from $\sim$4.5\,MOPS to $\sim$2,945\,ops/s ($\sim$9.8\,ms latency) --- a \textbf{1,500$\times$ slowdown} caused by memory swapping when the working set exceeds available DRAM.
  \item \textbf{Write degradation at scale:} YCSB-F throughput \emph{decreases} from 2.50\,MOPS (16T, 2CN) to 2.35\,MOPS at 32T --- a negative scaling pattern not reported in the paper, indicating contention on insert/update paths.
  \item \textbf{Coroutine overload:} Coro=3 at 32 threads causes latency to explode to 21\,$\mu$s (vs.\ 6\,$\mu$s for coro=1 at the same throughput), indicating no backpressure mechanism exists to prevent queue depth saturation.
  \item \textbf{CXL/NUMA resize stalls:} Resize events cause a $\sim$95\% throughput stall (near-zero MOPS for 1--12\,s) because the shared-memory remap blocks all clients atomically. The 16-thread 4th resize shows a 12-second zero-throughput window.
  \item \textbf{Range queries impossible:} YCSB-E cannot be run (see \S\ref{sec:codebase-issues}), exposing a fundamental limitation of the DMPH indexing approach for workloads requiring ordered access.
\end{enumerate}

\subsection{Sensitivity Analysis (Paper Figures~14--15)}
% ──────────────────────────────────────────────────────────────────────────────

\textbf{Load factor (Paper Fig.~14):} The paper reports that varying load factor from 0.75 to 0.95 causes ``trivial'' impact on throughput ($\sim$6\,MOPS for all). Our RDMA reproduction confirms this: FB throughput varies from 4.50 (LF=0.75) to 4.82 (LF=0.95) MOPS at 32T. Our CXL/NUMA results also confirm: 124.8 (LF=0.75) to 118.1 (LF=0.95) --- a 5.4\% difference.

\textbf{KV pair count (Paper Fig.~15):} The paper reports that increasing from 20M to 80M KV pairs decreases throughput from 6.02 to 5.83\,MOPS ($\sim$3.2\% decrease) on CX-6. Our RDMA reproduction at 32T shows: 4.49 (20M) to OOM crash (80M) --- the r320's 16\,GB DRAM is insufficient for 80M keys once the raw KV array, DMPH index, and pre-registered RDMA huge-page MR are all in memory. At 60M keys with LF=0.80, throughput collapsed to $\sim$2,945\,ops/s ($\sim$9.8\,ms latency), consistent with heavy paging/swap activity once the working set spills out of physical memory. Our CXL/NUMA results show: 149.3 (20M) to 100.6 (80M) --- a \textbf{32.6\%} decrease dominated by L3 cache misses.

Three key discrepancies emerge between the paper's claims and our measurements:
\begin{enumerate}[nosep]
  \item \textbf{Power-of-2 bucket allocation.} The seed array requires one byte per bucket, and bucket counts are powers of two. The Othello arrays are also sized proportionally to bucket count. When $n$ crosses a doubling boundary (e.g., 60M at LF=0.85 triggers $2^{24} \to 2^{25}$ buckets), the index size nearly doubles while both the paper's formula and Figure~16 predict smooth $\sim$50\% growth.
  \item \textbf{Consistent overprovisioning beyond the paper's data.} Even the paper's own Figure~16 measurements ($\sim$12.5\,MB at 20M) are $\sim$10\% above the theoretical formula (10.6--11.5\,MB). Our measurements (15.5--16.9\,MB at 20M) are a further 24--35\% above the paper's Figure~16, totaling 34--55\% above the formula. This gap is caused by power-of-2 sizing of Othello internal hash arrays and lock array overhead ($\sim$16\,KB per table).
  \item \textbf{Plateau behavior.} Because bucket counts are powers of two, the index size is \emph{identical} for 60M, 80M, and 100M keys at LF$\leq$0.85 (all use $2^{25}=33$M buckets). Neither the paper's formula nor its Figure~16 captures this plateau, which means adding 40M more keys (60M$\to$100M) costs \emph{zero} additional memory.
\end{enumerate}

Across both platforms, the DMPH index sizes are virtually identical: RDMA and CXL/NUMA differ by at most $\sim$16\,KB at LF$\leq$0.90 (the lock array is excluded from the CXL/NUMA port since it uses process-level mutexes instead). At LF=0.95, the CXL/NUMA Othello arrays are $\sim$3\% larger because the \texttt{client\_numa} code uses a slightly different rounding path for the array allocation at the highest load factor. This confirms that the DMPH data structure is identical across platforms --- the architectural difference lies solely in how the indexed data is \emph{accessed} (RDMA RPC vs.\ direct load/store), not in how the index is \emph{stored}.

In addition, the actual RDMA client \emph{process} RSS far exceeds the DMPH index: 364.6\,MB for 20M keys and 915.8\,MB for 60M keys (at LF=0.85), which is 23$\times$ and 14$\times$ the DMPH index, respectively. The CXL/NUMA client avoids this overhead entirely --- no RDMA buffer registration, no QP connection state, and no coroutine stacks are needed. The paper does not report process-level memory at all, presenting only the DMPH index as the compute-node memory cost.

\section{Literature Review}
% ══════════════════════════════════════════════════════════════════════════════

This section surveys the systems and techniques directly relevant to this work:
the Outback paper being reproduced, the RDMA-based KV store baselines compared in
all experiment graphs, the hashing primitives that underpin Outback's index, and
the CXL memory interconnect technology used for the extension.

% ──────────────────────────────────────────────────────────────────────────────
\subsection{The Reproduced Paper: Outback}
% ──────────────────────────────────────────────────────────────────────────────

\textbf{Outback}~\cite{outback} is the primary system studied in this report.
It addresses the \emph{one-sided vs.\ two-sided RDMA dilemma}: one-sided RDMA
requires multiple round trips for traditional hash or tree indexes; two-sided RPC
resolves the index in one round trip but imposes CPU load on memory nodes,
violating the disaggregation principle.  Outback resolves this by splitting the
index into two components:
\begin{itemize}[nosep]
  \item A \emph{compute-heavy, memory-efficient} DMPH seed component
        (Ludo~\cite{ludo} arrays) cached entirely on compute nodes.
  \item A \emph{memory-heavy, compute-efficient} Othello bucket-locator
        component residing on memory nodes and accessed with a single
        one-sided RDMA READ.
\end{itemize}
The prototype is evaluated on a CloudLab cluster using Mellanox ConnectX-6
100\,Gbps InfiniBand and achieves 1.06--5.03$\times$ throughput over the
state-of-the-art baselines (RACE, RPC-MICA, Cluster) on YCSB workloads.
Published as a full paper in \emph{PVLDB}, vol.\ 18, 2024.

% ──────────────────────────────────────────────────────────────────────────────
\subsection{Baseline Systems Compared in the Graphs}
% ──────────────────────────────────────────────────────────────────────────────

The experiment graphs in Sections~\ref{sec:repro-ycsb}--\ref{sec:repro-resize}
and in the three-way comparative analysis include three published baselines.

\textbf{RACE}~\cite{race} (\emph{One-Sided RDMA-Conscious Extendible Hashing}) is
an extendible hashing index designed for disaggregated memory~\cite{race}.  All
operations --- lookups, inserts, and resizing --- are performed via one-sided RDMA
verbs without involving memory-node CPUs. RACE uses a lock-free remote concurrency
control scheme and extendible remote resizing.  Outback's paper reports 2.43--5.03$\times$
throughput over RACE on YCSB-A/C; our RDMA reproduction corroborates the relative
ordering (5.04\,MOPS vs.\ the paper's 4.59\,MOPS for RACE at equivalent thread counts).
Published in \emph{USENIX ATC 2021}.

\textbf{MICA}~\cite{mica} (\emph{A Holistic Approach to Fast In-Memory Key-Value
Storage}) is a single-machine in-memory KV store tuned for multi-core CPUs with
kernel-bypass networking~\cite{mica}. It achieves 65--77\,MOPS on a single server
using partitioned data and a lightweight DPDK-based network stack. In Outback's
evaluation, MICA is invoked via two-sided RPC (\textit{RPC-MICA}), which adds
network round-trip overhead; it is the weakest baseline in the paper's
YCSB-C comparison (2.47\,MOPS at 72 CN threads).
Published in \emph{USENIX NSDI 2014}.

\textbf{Cluster RPC} is the third baseline reported in Outback's Figure~9.  It
represents a conventional cluster-hash approach (similar to the hash table in
DrTM+R~\cite{drtmr}) accessed via two-sided RPC.  It reaches $\sim$5.41\,MOPS at
72 threads --- slightly above RACE and slightly below Outback --- forming the
middle of the baseline band in all throughput-scaling figures.

% ──────────────────────────────────────────────────────────────────────────────
\subsection{Additional Related RDMA KV Stores in the Codebase}
% ──────────────────────────────────────────────────────────────────────────────

The repository also contains benchmark stubs for three additional RDMA KV
systems that appear in related work comparisons in the broader literature,
though they are not plotted as baselines in this report's graphs.

\textbf{FaSST}~\cite{fasst} (\emph{Fast, Scalable and Simple Distributed
Transactions}) uses two-sided unreliable-datagram RPCs over RDMA instead of
one-sided operations~\cite{fasst}.  It demonstrates that two-sided RPCs can
outperform one-sided RDMA in transactional workloads because of better
flexibility, while avoiding the scalability bottlenecks of RC-based one-sided
RDMA in large clusters.  Published in \emph{OSDI 2016}.

\textbf{DrTM+R}~\cite{drtmr} (\emph{Fast In-Memory Transaction Processing Using
RDMA and HTM}) combines RDMA with hardware transactional memory (HTM) to achieve
high-performance distributed ACID transactions~\cite{drtmr}.  The \texttt{r2}
communication library used by Outback is derived from DrTM+H, the successor
system presented at \emph{OSDI 2018}~\cite{drtmh}.

\textbf{ROLEX}~\cite{rolex} (\emph{A Scalable RDMA-oriented Learned Key-Value
Store for Disaggregated Memory Systems}) replaces perfect-hash indexes with
learned indexes (B-tree models) to reduce the number of RDMA round trips for
lookups~\cite{rolex}.  Like Outback, it processes all requests on compute nodes
via one-sided RDMA.  Published in \emph{USENIX FAST 2023}.

% ──────────────────────────────────────────────────────────────────────────────
\subsection{Key Algorithmic Primitives}
% ──────────────────────────────────────────────────────────────────────────────

\textbf{Ludo Hashing}~\cite{ludo} is the compact perfect-hashing scheme that
forms the compute-node component of Outback's DMPH index~\cite{ludo}. It achieves
$3.76 + 1.05\ell$ bits per key-value item (for $\ell$-bit values), which is near
the information-theoretic minimum. Ludo supports dynamic insertions and deletions
with low amortized cost.  Outback uses Ludo seed arrays to resolve the hash
function parameter for a query key before issuing the RDMA READ.
Published in \emph{ACM SIGMETRICS / POMACS 2020}.

\textbf{YCSB}~\cite{ycsb} (\emph{Yahoo! Cloud Serving Benchmark}) is the
standard benchmark suite used throughout this report~\cite{ycsb}. The six
workloads (A--F) span read-heavy, write-heavy, and scan-heavy access patterns,
and are parameterized by key distribution (uniform, Zipfian) and operation mix.
All throughput and latency figures in this report use 50\,M key-value pairs with
a 10\,M key working set under Zipfian distribution.
Published in \emph{ACM SoCC 2010}.

% ──────────────────────────────────────────────────────────────────────────────
\subsection{CXL Memory Disaggregation}
% ──────────────────────────────────────────────────────────────────────────────

\textbf{Compute Express Link (CXL)}~\cite{cxl} is an open interconnect standard
built on PCIe that provides cache-coherent, load/store-accessible remote memory.
CXL.mem enables host CPUs to access remote memory at 150--300\,ns latency ---
10--20$\times$ lower than RDMA's typical 2--5\,$\mu$s round-trip.  CXL~3.0
extends the standard to support multi-node memory pooling.  In this work, CXL.mem
semantics are emulated via NUMA inter-socket shared memory on a dual-socket
CloudLab c6420 machine, where cross-socket UPI latency ($\sim$100--200\,ns)
provides a faithful proxy.

% ══════════════════════════════════════════════════════════════════════════════
\section{Discussion}
% ══════════════════════════════════════════════════════════════════════════════

\subsection{CXL Advantages Over RDMA}

\begin{enumerate}[nosep]
  \item \textbf{Latency:} CXL/NUMA achieves 0.13--0.15\,$\mu$s mean latency vs.\ RDMA's 2--3\,$\mu$s --- a \textbf{15--20$\times$} improvement.
  \item \textbf{Scalability:} Near-linear scaling with thread count (throughput $\propto$ threads) because no shared NIC bottleneck exists.
  \item \textbf{Simplicity:} Eliminating the network stack removes coroutine schedulers, RPC layers, QP management, and memory registration --- reducing code complexity by $\sim$60\%.
  \item \textbf{Memory Efficiency:} No RDMA buffer registration overhead, connection state, or NIC-side memory consumption.
\end{enumerate}

\subsection{CXL Limitations}

\begin{enumerate}[nosep]
  \item \textbf{Single-Machine Scope:} Unlike RDMA (cluster-wide), CXL/NUMA is limited to one machine's socket topology. CXL~3.0 pooling is needed for multi-node.
  \item \textbf{Write Contention:} Without hardware offloading, concurrent writes compete on the memory bus. Software lanes mitigate but don't eliminate this.
  \item \textbf{Resize Coordination:} Shared-memory remap requires atomic generation counters; stale pointers risk segfaults.
  \item \textbf{Persistent Shared Memory:} \texttt{/dev/shm} files require explicit cleanup via signal handlers.
\end{enumerate}

% ══════════════════════════════════════════════════════════════════════════════
\section{Conclusion}
% ══════════════════════════════════════════════════════════════════════════════

This study reproduced Outback's RDMA experiments on nine CloudLab Apt \texttt{r320} nodes and extended the system to a CXL-emulated platform via NUMA inter-socket shared memory. Together, the two tracks answer a central question: \emph{how much of Outback's design is intrinsic to its DMPH index, and how much is an artifact of the RDMA transport?}

\textbf{RDMA reproduction.} Running on the same hardware class (Mellanox CX-3 56\,Gbps FDR InfiniBand) as the paper's Figure~10 cluster, we confirmed Outback's core architectural claim: a single-RTT DMPH lookup scales linearly with MN-thread count on read-dominated workloads (YCSB-C peak: 5.04\,MOPS at 4 MN threads, 64 total CN threads). The coroutine-depth sweep (Fig.~13) reproduced the expected hockey-stick curve, with coro=2 doubling throughput over coro=1 and coro=3 yielding diminishing returns. However, the reproduction exposed several undisclosed issues: write-heavy workloads (YCSB-A, YCSB-F) saturate and degrade at high thread counts; the resize experiment (Fig.~16--17) exhibits a $\sim$45\% throughput drop with erratic recovery rather than the claimed smooth $\sim$52\% degradation; the actual DMPH memory footprint (16.87\,MB) and process RSS (915.8\,MB) substantially exceed the paper's reported 12.5\,MB; and three out of four baseline systems (RACE, MICA, Cluster) fail to build, preventing direct ratio verification.

\textbf{CXL/NUMA extension.} Replacing the RDMA transport with POSIX shared memory across NUMA sockets yielded a \textbf{23$\times$} throughput gain (118\,MOPS at 32 threads vs.\ 5\,MOPS on RDMA) and a \textbf{21$\times$} latency reduction (0.14\,$\mu$s vs.\ 3\,$\mu$s mean). Thread scaling remained near-linear because the NUMA memory bus imposes no NIC bottleneck. The port required re-engineering the transport, removing the coroutine scheduler, building software allocation lanes to replace RDMA queue pairs, and implementing a versioned resize state machine with signal-safe \texttt{/dev/shm} cleanup --- reducing codebase complexity by approximately 60\% while improving all performance metrics by over an order of magnitude.

\textbf{Broader implications.} The results suggest that Outback's DMPH indexing strategy is architecturally sound and transport-agnostic: the same index design that enables single-RTT RDMA reads maps directly to single-load CXL accesses, with far better performance characteristics. As CXL.mem devices become commercially available, the disaggregated memory architecture Outback targets will shed the latency and complexity costs of RDMA, rendering the transport layer a bottleneck rather than an enabler. The primary constraint on RDMA-based disaggregation --- per-QP NIC state, hugepage memory registration, and multi-RTT coroutine scheduling --- is entirely absent in the CXL model. Our port demonstrates that this transition requires engineering effort but is structurally straightforward for DMPH-indexed systems.

% ── Limitations ────────────────────────────────────────────────────────────
\subsection{Limitations}

Several limitations constrain the conclusions drawn in this study:

\begin{itemize}[nosep]
  \item \textbf{NUMA as CXL proxy:} Our CXL results are produced on a dual-socket NUMA machine (CloudLab \texttt{c6420}) with inter-socket DDR4, not real CXL hardware. NUMA avoids PCIe bandwidth and latency constraints present in actual CXL devices; real CXL latency is typically 200--400\,ns (vs.\ our 130--150\,ns), and PCIe bandwidth caps ($\sim$32\,GB/s for PCIe 5.0 $\times$16) may limit high-thread throughput in ways NUMA does not.

  \item \textbf{Incomplete codebase:} The upstream Outback repository has build-breaking issues in its RACE, MICA, and cluster-hash baselines (see \S\ref{sec:codebase-issues}), preventing reproduction of the paper's Fig.~7--8 comparison ratios. Our throughput numbers cannot be placed in context against these baselines.

  \item \textbf{RDMA reproduction gap:} Our 9-node CloudLab cluster uses Mellanox CX-3 56\,Gbps FDR InfiniBand, identical to the paper's hardware. However, our aggregate throughput at peak (8T/CN) falls short of the paper's reported peak, likely due to differences in server-side DRAM allocation, hugepage configuration, and MLNX\_OFED driver versions that were not disclosed.

  \item \textbf{Resize experiment incomplete:} The paper's dynamic resizing experiment (Fig.~16) requires a stable server that survives the measurement window. The upstream resize code segfaults under load; our NUMA resize state machine is functional but produces data only for the NUMA transport, not the RDMA path.

  \item \textbf{Single-machine CXL scope:} CXL/NUMA is inherently limited to one physical host. The disaggregated memory pool model (multiple compute nodes sharing one memory node) cannot be evaluated without either CXL~3.0 fabric hardware or a multi-machine emulation layer.

  \item \textbf{Range query unsupported:} YCSB-E (range scans) cannot be evaluated because the DMPH index does not support ordered traversal, making scan-heavy workload comparisons impossible with the current design.
\end{itemize}

% ── Future Work ────────────────────────────────────────────────────────────
\subsection{Future Work}

\begin{itemize}[nosep]
  \item \textbf{Real CXL hardware validation.} Deploy Outback-CXL on a server equipped with a Samsung CXL memory expander (e.g., CMM-D 128\,GB) or Intel/Micron CXL DIMMs to replace the NUMA emulation and validate whether PCIe bandwidth or higher CXL latency (200--400\,ns vs.\ NUMA's $\sim$130\,ns) changes the throughput/latency trade-off reported here.

  \item \textbf{CXL~3.0 multi-node pooling.} Extend the prototype to a CXL fabric switch (e.g., Astera Labs Leo or Microchip CXL switch) to evaluate whether the single-machine scope limitation can be removed while retaining the latency advantage over RDMA.

  \item \textbf{Comparative evaluation against CXL-native KV stores.} Several recent systems target CXL disaggregated memory and serve as natural baselines:
    \begin{itemize}[nosep]
      \item \textbf{CHash}~\cite{chash}: A CXL-aware hash table that exploits byte-addressability and persistence on CXL-attached memory; directly comparable to Outback's DMPH index on a CXL device.
      \item \textbf{MIND}~\cite{mind2021}: Disaggregated memory with hardware-accelerated coherence; shows the overhead of maintaining cache coherence across CXL nodes.
      \item \textbf{AIFM}~\cite{aifm2020}: Application-integrated far memory using a user-space runtime; comparable to Outback's software-lane approach.
      \item \textbf{Carbink}~\cite{carbink2022}: Fault-tolerant disaggregated memory with erasure coding; relevant for evaluating the reliability cost of memory disaggregation.
    \end{itemize}

  \item \textbf{Write-heavy workload optimization.} YCSB-A and YCSB-F show the largest gap between RDMA and paper results. Investigate software write-combining, per-lane write buffers, and CLWB/CLFLUSHOPT instructions on CXL to reduce write contention on the memory bus.

  \item \textbf{Baseline reproduction.} Fix the broken RACE, MICA, and cluster-hash builds to reproduce the paper's Fig.~7--8 comparison ratios and place the CXL numbers in the same coordinate space as the original evaluation.

  \item \textbf{CXL resize under contention.} Extend the versioned resize state machine to handle concurrent readers during expansion, and measure resize latency spikes under the YCSB workloads to validate the paper's claim of non-disruptive resizing.
\end{itemize}
